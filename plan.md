# BIS Standards Recommendation Engine — Build Plan

## What We Are Building

A RAG (Retrieval-Augmented Generation) pipeline that:
1. Ingests BIS SP 21 (2005) PDF — 929 pages of building material standard summaries
2. Accepts a product description query
3. Returns top 3–5 relevant BIS standard IDs with rationale
4. Exposes a Streamlit UI
5. Runs `inference.py` for judge evaluation
6. Passes `eval_script.py` without modification

---

- No vector DB install needed — numpy cosine similarity over ~500 chunks is fast enough
- `pymupdf` (fitz) is already installed

## Step 1 — PDF Parsing & Chunking (`src/ingest.py`)

### PDF Structure (observed from dataset.pdf)
- Pages 1–12: Front matter, table of contents — skip
- Page 13 onward: Actual standard summaries
- Each standard occupies 1–3 pages
- Each standard page starts with a header like:
  `SUMMARY OF\nIS 269 : 1989 33 GRADE ORDINARY PORTLAND CEMENT`
- The IS number pattern is: `IS\s+\d+[\w\s():.]*:\s*\d{4}`

### Chunking Strategy
- **Per-standard chunking**: Each chunk = one complete standard summary
- Parse all pages from page 13 onward
- Detect standard boundaries using the regex: `SUMMARY OF\s+IS\s+[\d]`
- Accumulate text until the next `SUMMARY OF` header
- Each chunk stores:
  ```python
  {
    "standard_id": "IS 269: 1989",       # normalized from header
    "title": "33 Grade Ordinary Portland Cement",
    "text": "full summary text...",       # used for embedding
    "page_start": 22                      # for debugging
  }
  ```
- **Fallback**: If a page has no `SUMMARY OF` header but has content, append it to the previous chunk (handles multi-page standards)

### Edge Cases in Parsing
- Standard IDs with parts: `IS 1489 (Part 1): 1991` — keep full string including part
- Year variants: some have spaces around colon, some don't — normalize to `IS XXXX: YYYY`
- Table of contents pages (pages 14–16) also contain IS numbers — skip them (they appear before page 17 where actual summaries start)
- Pages with only whitespace or roman numerals (i, ii, iii...) — skip
- Multi-page standards: pages that continue a standard without a new `SUMMARY OF` header — append to current chunk

### Embedding
- Use Gemini `text-embedding-004` model
- Embed `title + "\n" + text[:2000]` per chunk (truncate to stay within token limits)
- Save all chunks + embeddings as `index/chunks.pkl` using pickle
- This runs once; inference.py loads the pre-built index

## Step 2 — Retriever (`src/retriever.py`)

- Load `index/chunks.pkl` at startup
- For a query string:
  1. Embed the query using Gemini `text-embedding-004`
  2. Compute cosine similarity between query embedding and all chunk embeddings (numpy dot product on L2-normalized vectors)
  3. Return top-K chunks (K=5) sorted by score
- Return list of `{"standard_id", "title", "text", "score"}`

### Edge Cases
- If index file missing: raise clear error message telling user to run `ingest.py` first
- If Gemini API key missing: raise with instructions to set `GEMINI_API_KEY` env var
- Tie-breaking: stable sort by standard_id string when scores are equal

## Step 3 — RAG Pipeline (`src/pipeline.py`)

```
query → embed → retrieve top-5 → build prompt → Groq LLM → parse output
```

### Prompt Design
```
You are a BIS standards expert. Given a product description, identify the most relevant standards from the retrieved context.

Product Description: {query}

Retrieved Standards:
{for each chunk: "IS XXXX: YYYY - Title\nSummary: first 300 chars..."}

Return ONLY a JSON array of objects with keys "standard_id" and "rationale".
List only standards that are genuinely relevant. Maximum 5, minimum 3.
Do not invent any standard not listed above.
```

- Use `llama-3.3-70b-versatile` on Groq (fast, free)
- Temperature = 0 for deterministic output
- Parse JSON from LLM response; if JSON parse fails, fall back to returning the top-3 retrieved standard IDs with no rationale (prevents hallucination)

### Anti-Hallucination Guardrails
- The prompt explicitly says "only from the list above"
- After LLM response, validate every returned `standard_id` against the retrieved set — strip any that weren't in the retrieved context
- If LLM returns fewer than 3 valid standards, pad with next highest-scoring retrieved standards

### Latency
- Gemini embedding: ~0.3–0.5s
- Groq LLM: ~0.5–1.5s
- Numpy cosine search: <0.01s
- Total expected: ~1–2s per query, well under 5s target

## Step 4 — `inference.py` (Judge Entry Point)

Exact interface required by rulebook:
```
python inference.py --input hidden_private_dataset.json --output team_results.json
```

### Input format (from public_test_set.json):
```json
[{"id": "PUB-01", "query": "...", "expected_standards": [...]}]
```

### Output format (from sample_output.json):
```json
[
  {
    "id": "PUB-01",
    "query": "...",
    "expected_standards": [...],
    "retrieved_standards": ["IS 269: 1989", "IS 8112: 1989", ...],
    "latency_seconds": 1.24
  }
]
```

### Logic
1. Load input JSON from `--input`
2. For each item, run the pipeline, measure wall-clock latency
3. `retrieved_standards` = list of standard_id strings, top-5, in ranked order
4. Write output JSON to `--output`

### Edge Cases
- Input file not found: print clear error and exit with code 1
- Individual query failure (API error, timeout): catch exception, write empty `retrieved_standards: []` and `latency_seconds: 0.0` for that item, continue processing rest
- Output directory doesn't exist: create it
- Malformed input JSON: print error and exit with code 1
- `expected_standards` key may or may not be present in hidden test set — always include it if present, omit if absent (do not crash)

## Step 6 — `eval_script.py` Compatibility

The provided `eval_script.py`:
- Reads a JSON file via `--results` argument
- Expects keys: `expected_standards`, `retrieved_standards`, `latency_seconds`
- Normalizes standard IDs by removing spaces and lowercasing

**Critical**: Our `inference.py` output must include `expected_standards` (pass-through from input) so `eval_script.py` can compare. The `retrieved_standards` values must match the normalization — e.g., `"IS 269: 1989"` normalizes to `"is269:1989"` which must match `"IS 269: 1989"` from expected. This works as long as we store standard IDs consistently.

## Known Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Gemini free tier rate limit (1500/day) | Index is built once and cached; only query embedding hits API at inference time |
| Groq rate limit | Free tier allows ~30 req/min; more than enough for evaluation |
| Standard ID format mismatch in eval | Normalize all stored IDs to `IS XXXX: YYYY` format at ingest time; eval_script normalizes by removing spaces anyway |
| LLM hallucinating a non-existent standard | Post-process: validate every returned ID against retrieved set; strip invalid ones |
| Multi-part standards (e.g. IS 1489 Part 1 vs Part 2) | Each part is a separate chunk with its own embedding; retriever returns the most relevant part |
| Table of contents pages containing IS numbers | Skip pages before the first `SUMMARY OF` header |
| Hidden test set queries outside building materials | Pipeline still works — retriever returns best match from available corpus |
